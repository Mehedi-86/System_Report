	\chapter*{Abstract}
	\addcontentsline{toc}{chapter}{Abstract}
	
	RestrictedIDE-V2 is a secure, policy-driven examination Integrated Development Environment (IDE) designed to support controlled programming assessments in both standalone and LAN-based lab settings. Traditional online coding platforms often depend on cloud connectivity and provide limited control over host-level behavior, which creates challenges for institutions that require strict invigilation, offline operation, and local policy enforcement. This project addresses those challenges by combining a lightweight frontend IDE with a Rust/Tauri backend that can enforce execution and interaction constraints.
	
	The system uses a modular architecture with a JavaScript-based editor interface and a command-oriented Rust service layer. Core capabilities include session creation and join workflows, timed assessments, controlled code execution, role-based administration, and event logging. A policy engine validates operations such as file access, process invocation, timing, and user actions before command execution. A local SQLite store maintains sessions, participants, submissions, violations, and broadcast events, while a LAN server enables multi-user assessment in offline campus networks.
	
	Security features in the current implementation include focus monitoring, clipboard/screenshot guarding hooks, and keyboard/process restrictions (platform-dependent), supported by centralized configuration and auditable logs. Experimental usage in project-level test scenarios shows that the platform can provide stable exam orchestration, deterministic submission flows, and improved control compared with a generic IDE setup. RestrictedIDE-V2 demonstrates a practical approach to building a controlled coding assessment platform with extensible policy and runtime boundaries.
	
	This report documents the full project lifecycle in detail: requirement engineering, architectural decomposition, policy design, implementation strategy, validation methodology, and ethical/societal analysis. In addition to describing the technical decisions, the report includes module-level explanations, operational workflows, simulated performance observations, risks, limitations, and a comprehensive roadmap for future development. The objective is not only to present a working artifact, but also to establish a repeatable engineering reference for secure coding assessment platforms in academic institutions.
	
	\textbf{Keywords:} RestrictedIDE-V2, secure examination IDE, policy enforcement, Rust/Tauri desktop architecture, offline LAN assessment, session lifecycle management, timed submission, academic exam invigilation.

	\chapter{Introduction}
	
	\section{Introduction}
	Programming assessments demand a reliable environment that is fair, secure, and easy to administer. In conventional lab examinations, students may rely on unrestricted local tools, open browser tabs, or unauthorized resources, which reduces assessment integrity. RestrictedIDE-V2 is proposed as a focused examination IDE to enforce policy constraints while preserving usability for students and instructors.
	
	In many educational contexts, the practical challenge is not whether programming can be evaluated, but whether the evaluation can be conducted in a controlled, transparent, and repeatable environment. Traditional methods, such as manually collecting source files through external storage devices, lead to inconsistent timing, missing submissions, and weak traceability. Web-only platforms solve some of these concerns, yet they may fail when internet connectivity is unstable or when organizations require on-premise deployment for compliance reasons.
	
	RestrictedIDE-V2 addresses this operational gap by unifying the coding environment, session orchestration, security policy checks, and submission management in one desktop application. The solution follows a local-first architecture with optional LAN communication, allowing institutions to run assessments in isolated laboratory networks. In essence, the project aims to create a practical bridge between strict invigilation requirements and developer-friendly exam workflows.
	
	The project is also motivated by pedagogical fairness. In a traditional practical exam, students with faster internet, better personal tooling, or informal access to external resources may gain unintended advantages over others. A restricted and standardized environment ensures that performance is measured primarily by algorithmic reasoning, coding quality, and problem-solving skill. Therefore, the software environment itself becomes part of assessment design, not just a passive medium.
	
	\section{Background / Problem Statement}
	Institutions that run programming exams often face the following operational problems:
	\begin{itemize}
		\item Lack of offline-capable assessment infrastructure.
		\item Weak host-level controls for application and process usage.
		\item Manual exam management and submission collection.
		\item Poor auditability of policy violations and exam events.
	\end{itemize}
	RestrictedIDE-V2 addresses these challenges through local-first architecture, session management, structured policy checks, and centralized records.
	
	From a systems perspective, the problem can be reframed as a constrained execution environment challenge. Students require sufficient freedom to write, run, and debug code; however, the examination authority requires constraints that prevent external assistance, unauthorized resource usage, and post-deadline manipulation. Designing a platform that satisfies both demands requires careful boundary control across UI behavior, file operations, process lifecycle, timing semantics, and communication channels.
	
	The project therefore defines the following core problem statement: \textit{How can a desktop IDE be transformed into an assessment-grade platform that enforces institutional policies while retaining acceptable usability and maintainability?} This statement drives all subsequent architectural and implementation decisions described in this report.
	
	The implementation and architectural choices are also aligned with established software engineering and secure coding principles discussed in foundational literature~\cite{sommerville,securecode}.
	
	To answer this, the work adopts a systems-engineering lens where exam integrity is treated as a composition of smaller guarantees: identity and role verification, deterministic timing, bounded execution, controlled storage access, and comprehensive auditability. If any one of these guarantees is missing, the overall trustworthiness of the exam can be compromised.
	
	\section{Objectives}
	The primary objectives of this project are:
	\begin{itemize}
		\item Design a secure IDE for controlled coding assessments.
		\item Implement exam session creation, join, and dashboard workflows.
		\item Enforce configurable policies for file/process/time/access control.
		\item Support timed submissions and violation monitoring.
		\item Provide maintainable architecture for future extension.
	\end{itemize}
	
	In addition to these primary objectives, the project also pursues several secondary targets:
	\begin{itemize}
		\item Ensure modular separation so that policy logic can evolve without redesigning the entire application.
		\item Provide auditable events for post-exam analysis and dispute resolution.
		\item Keep deployment lightweight enough for ordinary academic lab machines.
		\item Support maintainable code organization for future contributors.
	\end{itemize}
	
	Collectively, these objectives ensure that RestrictedIDE-V2 is not merely a demonstration prototype but a practically deployable academic system. Each objective is intentionally measurable through scenario-based validation, such as successful timed submissions, policy violation logging, and session reproducibility under repeated trial runs.
	
	\section{Scope of the Work}
	The project covers frontend editor integration, backend command orchestration, policy enforcement, database operations, and LAN session transport. It includes desktop deployment through Tauri and practical tools for exam administration. External cloud synchronization and cross-platform parity for all low-level controls are considered future extensions.
	
	The scope intentionally excludes advanced plagiarism detection engines, biometric proctoring, and cloud-scale container orchestration. Those features can be integrated later, but they are not required to validate the core thesis of this work. By constraining scope, the team prioritized architecture stability, predictable operation, and incremental extensibility.
	
	\section{Unfamiliarity of the Problem / Novelty}
	Unlike generic IDE wrappers, RestrictedIDE-V2 combines assessment workflow management with policy-aware runtime control in a single packaged desktop app. The novelty is in integrating editor usability, local transport, and enforcement modules under an extensible command architecture suitable for institutional labs.
	
	The specific novelty of RestrictedIDE-V2 may be summarized in three engineering contributions:
	\begin{enumerate}
		\item \textbf{Policy-guarded command execution:} all critical runtime actions are routed through backend commands that can be validated by centralized policy rules.
		\item \textbf{Offline-capable session orchestration:} exam workflow remains operational even without internet connectivity, using local storage and LAN transport.
		\item \textbf{Security-aware desktop UX:} the IDE interface is optimized for exam flow while integrating restrictions such as timed submissions and monitored events.
	\end{enumerate}

	\section{Introduction}
	Secure coding assessments are commonly delivered through cloud judges, LMS plugins, or locked browser environments. These systems usually emphasize result correctness and plagiarism detection but provide limited local runtime control.
	
	\section{Related Works}
	Existing online judge systems such as HackerRank-style platforms provide strong evaluation pipelines but depend on internet connectivity and remote sandbox infrastructures. Lockdown browser systems focus on web restrictions rather than full IDE workflows. Desktop exam systems often lack modular policy engines and do not combine session, runtime, and admin control in one local framework.
	
	Several literature streams are relevant to this project:
	\begin{itemize}
		\item \textbf{Online judge platforms:} highly scalable, centralized evaluation systems with problem bank management and automatic verdicting.
		\item \textbf{Secure browser ecosystems:} controlled exam interfaces for web-based assessments that restrict navigation and clipboard actions.
		\item \textbf{Desktop sandbox research:} host-level restrictions for process/file isolation, often used in enterprise endpoint security.
		\item \textbf{Hybrid assessment tools:} systems that combine local coding and periodic server synchronization.
	\end{itemize}
	
	Most of these systems optimize one dimension strongly (for example, internet-scale evaluation) but do not fully satisfy local institutional constraints, particularly in environments where reliable internet or cloud compliance pathways are unavailable.
	
	Another recurring limitation in existing tools is the lack of explainable policy outcomes. Many systems silently block actions or produce generic errors, which confuses users during high-stakes exams. RestrictedIDE-V2 emphasizes explicit policy-driven decisions and event records so that both students and administrators can interpret system behavior after an incident.
	
	From an access-control perspective, the role separation assumptions of this project follow the rationale of RBAC models~\cite{rbac}.

	\section{Discussion}
	RestrictedIDE-V2 is positioned between cloud judges and basic lockdown tools. It emphasizes local operation, extensible policies, and exam-specific desktop workflow control. This balance makes it suitable for universities with constrained network conditions.
	
	The comparative analysis suggests a practical trade-off: while cloud-native systems may offer richer evaluation ecosystems, a controlled desktop platform offers greater autonomy, infrastructure predictability, and direct policy governance for physical lab examinations. This positioning is aligned with the needs identified during requirement analysis.

	\section{Detailed Methodology}
	The development process followed an incremental modular approach:
	\begin{itemize}
		\item Requirement analysis from exam administration use cases.
		\item Separation of concerns across frontend, command layer, policy engine, and runtime/security modules.
		\item Iterative integration and functional verification.
	\end{itemize}
	
	Methodological decisions were guided by four principles:
	\begin{itemize}
		\item \textbf{Traceability:} every major feature maps to a documented requirement.
		\item \textbf{Isolation:} modules communicate through clear interfaces to reduce side effects.
		\item \textbf{Policy-first security:} sensitive actions are validated before execution.
		\item \textbf{Operational realism:} testing scenarios mirror actual lab exam constraints.
	\end{itemize}
	
	The methodology intentionally prioritized iterative verification over late-stage integration. Instead of waiting until all modules were complete, each subsystem was integrated with a minimal executable workflow and then expanded. This reduced defect propagation and made root-cause analysis significantly easier.
	
	\subsection{Problem Design and Analysis}
	The system was modeled around three actor groups: admin, student, and runtime enforcer. Data entities were derived as sessions, participants, submissions, violations, and broadcasts.
	
	Use-case analysis produced the following critical interaction paths:
	\begin{itemize}
		\item Admin authenticates and creates session with timing and policy profile.
		\item Student joins by session code and receives exam state.
		\item Student performs coding and optional local execution under guard.
		\item Timer expiry or manual submission triggers final submission record.
		\item Admin reviews submissions, violations, and completion status.
	\end{itemize}
	
	Threat modeling identified misuse vectors such as path traversal attempts, unauthorized process launching, session overrun manipulation, and policy bypass through direct command execution. These threat vectors directly influenced command gateway design.
	
	In design workshops, each threat vector was mapped to one or more enforcement points in the architecture. For example, unauthorized file access was addressed by backend path validation rather than frontend restrictions alone, because UI-only controls are easier to bypass. This mapping strategy ensured that controls were attached to trustworthy execution boundaries.

	\subsection{Core Algorithmic Logic}
	Policy evaluation follows a guard-before-execution approach. For each command request, applicable rules are checked first; if any rule fails, the operation is blocked and an event is logged.
	
	The policy-check sequence can be represented by the following pseudo-process:
	\begin{enumerate}
		\item Identify command category and associated policy domains.
		\item Resolve contextual metadata (user role, session state, path/process target, timestamp).
		\item Evaluate deterministic rules in predefined order.
		\item If all checks pass, execute command and persist outcome.
		\item If any check fails, reject request and persist violation with reason code.
	\end{enumerate}
	
	Let $R = \{r_1, r_2, \ldots, r_n\}$ denote rule predicates for a command context $c$. Execution is permitted only if:
	\[
	\forall r_i \in R,\ r_i(c)=\text{true}
	\]
	If at least one rule returns false, command state transitions to denied and an audit event is written.

	\section{Testing Methodology}
	Testing combined unit-level verification, command-level integration checks, and scenario-driven workflow trials.
	
	\subsection{Functional Tests}
	Functional tests validated each major user flow: session creation, join, coding state retention, submission, and dashboard retrieval.
	
	\subsection{Policy Tests}
	Policy tests attempted disallowed actions (invalid paths, restricted process names, and deadline bypass attempts) to verify deterministic rejection and logging behavior.

	Basic stress checks used repeated join/leave sequences and rapid submission attempts near timeout boundaries to verify consistent state transitions.

	The methodology ensured traceability from requirements to implementation while preserving maintainability through module isolation.

	\chapter{Implementation, Results and Discussions}
	
	\section{Introduction}
	This chapter presents the implemented modules and discusses system behavior under practical test conditions.
	
	Implementation followed the architectural design from Chapter III and focused on clear responsibility boundaries between frontend orchestration and backend enforcement. This chapter provides module-level details to demonstrate how design decisions were translated into executable components.
	
	\section{Frontend Implementation Details}
	The frontend is built around a lightweight modular JavaScript structure. Primary responsibilities are editor interaction, file and tab management, session workflow visualization, timer rendering, and user-triggered command dispatch.
	
	The frontend design favors predictable interaction flow over feature overload. During an exam, cognitive load should be devoted to solving programming problems rather than navigating the interface. Therefore, key actions are intentionally linear and context-aware.
	
	\subsection{Editor and Workspace Interaction}
	The editor subsystem integrates Monaco-based editing and local file handling abstractions. Key capabilities include syntax-aware editing, controlled save/open operations, and code execution triggers through backend command calls.
	
	A notable design choice is separating editing convenience from execution authority. Students can edit fluidly, but execution-affecting actions are validated at backend boundaries. This avoids coupling usability features with security-critical logic.
	
	\subsection{Session Screens}
	Session modules provide separate views for create, join, dashboard, question handling, post-session review, and submission flow. This segmentation supports maintainability and role separation.
	
	\subsection{Timer and Auto-Submission Workflow}
	A client-visible timer is synchronized with backend timing boundaries. On expiry, submission is automatically triggered to prevent post-deadline edits.
	
	\section{Backend Implementation Details}
	The backend is implemented in Rust using Tauri command interfaces and dedicated modules for policy, runtime, security, transport, and persistence.
	
	Rust was selected because of its strong type system, memory safety guarantees, and reliability under concurrent event flows. For an exam system where deterministic behavior is crucial, language-level safety contributes directly to operational confidence.
	
	Platform capabilities and implementation boundaries were guided by official ecosystem documentation for Tauri, Rust, Monaco, and SQLite~\cite{tauri,rust,monaco,sqlite}.
	
	\subsection{Command Modules}
	Command handlers act as the authoritative entry points. Each handler validates input, checks session context, invokes policy evaluation, and routes permitted operations to domain services.
	
	\subsection{Policy Engine Integration}
	Policy checks are encapsulated into focused rule modules (file, process, keyboard, URL, time). A command can map to one or more policies depending on operation risk.
	
	\subsection{Session and Transport Service}
	LAN transport exposes exam endpoints for controlled local-network access. Session state is synchronized through persistent storage and in-memory state transitions where necessary.
	
	\subsection{Persistence Layer}
	SQLite with relational tables supports deterministic query patterns and easy local deployment. The schema design prioritizes exam traceability and audit retrieval.
	
	\section{Security and Runtime Controls}
	Security modules include process monitoring, focus/watchdog components, and screenshot/interaction guards where platform support is available. Runtime modules coordinate exam mode transitions and restricted execution boundaries.
	
	These controls are treated as layered safeguards rather than a single absolute barrier. Even if one mechanism is limited by platform capabilities, policy checks and event logging still contribute to practical deterrence and forensic visibility.
	
	\section{Experimental Setup}
	The system was tested on desktop environments with Tauri runtime and local SQLite storage. Both single-machine and LAN-connected scenarios were used for validation.
	
	Test setup included:
	\begin{itemize}
		\item Isolated lab-style LAN with one admin machine and multiple student clients.
		\item Repeated 30--120 minute timed sessions.
		\item Mixed programming tasks requiring file edits and controlled code runs.
		\item Simulated policy breach attempts to validate response consistency.
	\end{itemize}
	
	\section{Evaluation Metrics}
	The following metrics were considered:
	\begin{itemize}
		\item Session setup success rate.
		\item Average join latency in LAN mode.
		\item Submission completion before timeout.
		\item Violation event detection consistency.
	\end{itemize}
	
	Additional qualitative indicators were recorded:
	\begin{itemize}
		\item Admin effort required to start and monitor an exam.
		\item Student clarity of session workflow and submission confidence.
		\item Stability of logs and post-session trace reconstruction.
	\end{itemize}

	\chapter{Security, Compliance, and Operational Considerations}
	
	\section{Security Architecture Considerations}
	RestrictedIDE-V2 applies a defense-in-depth strategy where sensitive operations are guarded at multiple layers. The frontend does not directly execute privileged actions; instead, it requests backend command handlers that perform policy checks before execution. This design reduces trust on client behavior and centralizes enforcement logic.
	
	From an implementation standpoint, the command-gateway model ensures that input validation, authorization checks, and audit logging happen in one deterministic flow. The same design also improves maintainability because security changes can be introduced in policy modules without rewriting UI components.
	
	\section{Threat Model and Attack Surface}
	The main attack vectors considered in this system include unauthorized file access, misuse of process execution, session deadline bypass attempts, identity misuse, and log tampering risk. Threat analysis indicates that exam systems are primarily vulnerable through boundary weaknesses, not through algorithmic complexity.
	
	To reduce attack surface, the design follows explicit allow/deny policy semantics, bounded path checks, role-sensitive command behavior, and time-based state constraints. These controls are complemented by event logs to support post-exam forensic review.
	
	\section{Policy Enforcement Strategy}
	Policy evaluation is modeled as a deterministic gate. Each operation is mapped to one or more policy families (time, file, process, URL, and interaction). If any required check fails, the operation is blocked and an explanatory violation record is created.
	
	This approach is important for technical transparency. Rather than silently rejecting actions, the system records policy outcomes and makes operator diagnostics easier during exam sessions.
	
	\section{Reliability and Fault Tolerance}
	In timed assessments, reliability is as important as security. Submission logic therefore prioritizes deterministic state transitions at deadline boundaries. A practical reliability strategy includes pre-submission buffering, timeout-safe transaction handling, and clear fallback behavior during unexpected runtime interruption.
	
	Operational resilience also depends on predictable recovery steps for invigilators. The deployment runbook should define recovery procedures for machine restart, delayed network events, and partial session interruptions.
	
	\section{Compliance and Auditability}
	A technical system report must address governance requirements in addition to implementation details. RestrictedIDE-V2 supports compliance by preserving event history for session creation, participant activity, submissions, and policy violations.
	
	Compliance readiness depends on institutional policy alignment for retention duration, access control, audit access permissions, and post-exam archival handling. The architecture is compatible with such governance workflows because logs and submission records are persisted in structured forms.
	
	Security baseline discussion in this report is additionally informed by widely accepted guidance from OWASP and secure architecture references~\cite{owasp,bishop}.
	
	\section{Operational Risk Management}
	Common operational risks include policy misconfiguration, inconsistent machine setup, and insufficient pre-exam verification. To control these risks, deployment procedures should include mandatory validation steps and a pre-exam checklist.
	
	Risk mitigation in this project is therefore not limited to code-level safeguards. It includes process-level safeguards such as operator training, session dry-runs, and standardized post-exam export procedures.

	\section{Complex Engineering Problems Associated with the Project}
	The project addresses a multi-constraint systems problem involving security, usability, reliability, offline operation, and cross-platform desktop integration.
	
	This multi-constraint nature qualifies the project as a complex engineering challenge because improvements in one axis may degrade another. For example, stronger restrictions can reduce usability; richer flexibility can weaken guarantees. The solution therefore requires explicit trade-off management.
	
	\subsection{Complexity Dimension 1: Security vs Usability}
	Policies must be strict enough to prevent misuse but predictable enough to avoid interrupting legitimate student workflow.
	
	\subsection{Complexity Dimension 2: Offline Capability vs Synchronization}
	Offline-first operation improves resilience but introduces challenges in post-session aggregation and optional remote synchronization.
	
	\subsection{Complexity Dimension 3: Platform Variability}
	Host-level controls may behave differently across operating systems, requiring adaptive abstractions and clear capability documentation.

	\chapter{Conclusions}
	
	\section{Summary}
	RestrictedIDE-V2 provides a practical secure examination IDE architecture with modular policy enforcement, session orchestration, and local-first runtime capabilities.
	
	The project demonstrates that exam governance can be built into the software execution model instead of relying solely on external invigilation procedures. By using a layered architecture and command-guarded design, the system remains maintainable while achieving strict operational control.
	
	Overall, the work validates that meaningful exam integrity improvements are achievable even within limited institutional infrastructure, provided that system design emphasizes deterministic operation, transparent policy enforcement, and maintainable implementation boundaries.
	
	\section{Limitations}
	Some security controls are platform-dependent, and advanced execution isolation can be improved with stronger OS-level sandboxing.
	
	Current evaluation scenarios are institution-scale and should be extended to larger and more heterogeneous environments. Automated regression testing infrastructure should also be expanded.
	
	\section{Recommendations and Future Works}
	Future development should include:
	\begin{itemize}
		\item Cross-platform parity for low-level lockdown controls.
		\item Stronger process isolation and resource quotas.
		\item Optional secure online synchronization and analytics.
		\item Enhanced proctor dashboard with richer violation forensics.
	\end{itemize}
	
	Additional recommended directions are:
	\begin{itemize}
		\item Declarative policy configuration profiles for course-specific rules.
		\item Optional plagiarism-assistance hooks with privacy-preserving modes.
		\item Advanced analytics for exam quality review and intervention planning.
		\item Accessibility and localization enhancements for diverse users.
	\end{itemize}

	Monitor participant count, watch violation summaries, and issue broadcast messages when required. Record exceptional events in an invigilator log for post-exam reconciliation.

	Close session, archive submissions and logs, and export summary reports for academic records. Verify archival integrity before ending the operational cycle.

	Backend modules are grouped by command domain, policy rules, security controls, session runtime, and persistence. This structure aligns implementation with responsibility boundaries and enables controlled extension.
	
	\subsection{Policy Modules}
	The policy layer isolates validation decisions from command business logic. This ensures consistent enforcement outcomes and reduces duplicated validation code across handlers.
